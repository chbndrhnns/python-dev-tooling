Before the quiz — five core problems every package manager has to solve. Expand each to understand the requirement, why it's hard, and how each tool approaches it.

Every package has its own dependencies, and those have dependencies too. The resolver's job is to find one version of every package such that all constraints are simultaneously satisfied — a problem that is NP-complete in the general case.

Version constraints are expressed as requests>=2.28,<3 (PEP 440). The subtle trap: a constraint that looks valid in isolation can become impossible once you add a second package that pins a conflicting range. A good resolver tells you why it failed and which packages caused the conflict — a bad one just says "no solution found".

The difference between a lock file and a requirements.txt matters here. A lock file is the output of resolution — every package, every transitive dependency, at an exact pinned version, with a hash. A hand-written requirements file is just a set of input constraints; two developers running it on different days may get different resolved versions.

lock file excerpt — uv.lock
# pinned by the resolver, not hand-written
[[package]]
name = "requests"
version = "2.31.0"
requires-dist = ["certifi>=2017.4.17", "charset-normalizer<4,>=2", ...]
wheels = [{url = "https://files.pythonhosted.org/...", hash = "sha256:58cd2187..."}]
pip + venvResolves correctly since pip 21 (resolvelib). No lock file — pip freeze is a manual snapshot, not a generated artifact.
PoetryStrong resolver with good conflict messages; generates poetry.lock. Slow on large trees due to eager metadata fetching.
uvPubGrub-derived algorithm (same family as Cargo). Parallel metadata fetching, aggressive caching. Clear, structured conflict output.
condaUses a SAT solver (libsolv). Correct but notoriously slow, and conflict messages are cryptic. environment.yml is not a full lock file.

Runtime isolation (venv) and build isolation (PEP 517/518) solve different problems but are often confused.

A virtual environment is a copy (or symlink tree) of a Python interpreter plus a private site-packages directory. Installing into it doesn't touch your system Python. The mechanism is simple: Python looks for packages in sys.prefix, and a venv changes what sys.prefix points to. venv is in the stdlib since Python 3.3; virtualenv is the older third-party tool that also supports Python 2.

Build isolation is different. When you run pip install . on a source distribution, pip needs to run the build system (e.g. setuptools, flit, hatchling) to produce a wheel. Without isolation, the build system uses whatever is already installed in your env — including version mismatches. PEP 518 introduced [build-system] in pyproject.toml, which tells pip to install build dependencies into a fresh temporary environment first.

pyproject.toml — build isolation declaration
[build-system]
requires    = ["hatchling"]   # installed into isolated build env
build-backend = "hatchling.build"
pip + venvYou create and activate the venv manually. Build isolation supported but not the default — needs --isolated or a [build-system] table.
PoetryCreates a project-scoped venv automatically on first run. PEP 517 build isolation on by default.
uvuv sync creates the venv and installs in one step. Build isolation is always on.
condaconda envs are heavier — they contain a full interpreter + all non-Python deps. Not venvs; don't use PEP 517. conda-build is a separate tool.

A package's runtime dependencies (what end users need) are different from its development dependencies (pytest, mypy, black) and optional extras (docs, async support). Conflating them causes two problems: end users get bloated installs, and CI environments become harder to reason about.

Extras (PEP 508: package[extra]) are optional runtime features the user can opt into. Dependency groups (first-class in Poetry, standardised in PEP 735 for pyproject.toml) are sets of packages that are never shipped — they only exist in the development environment.

Publishing is the other side of the same coin: once your project is built, it needs to be packaged as a wheel + sdist and uploaded to PyPI (or a private index). This requires building the distribution artefacts and authenticating against the registry — steps that pip alone doesn't cover.

Poetry — groups

[tool.poetry.group.dev.dependencies] — installed only in dev, never packaged. poetry publish handles build + upload in one command.

uv — groups

[dependency-groups] in pyproject.toml (PEP 735). uv publish added in 0.4. uv build produces wheel + sdist.

pip — no groups

No native concept. Convention: multiple files (requirements-dev.txt). Extras are supported. Publishing requires separate build + twine invocations.

conda — different model

conda packages go to Anaconda or conda-forge — a completely separate infrastructure from PyPI. No concept of dev groups in the Python sense.

Performance in package management has two distinct bottlenecks: network I/O (fetching package metadata and wheels from PyPI) and resolver computation (the backtracking search). In practice, network I/O dominates for most real-world installs.

pip fetches metadata sequentially — it picks a candidate, fetches its metadata, looks at its dependencies, fetches those, and so on. For a project with 50 transitive dependencies this can mean 50+ serial HTTP round trips. uv parallelises all of these, and caches the results in a global wheel cache keyed by hash — a subsequent install of the same package anywhere on the machine is a local copy, no network involved.

The resolver algorithm also matters. pip's resolvelib can do significant backtracking on complex version ranges. uv's PubGrub implementation bounds the backtracking more tightly, and Poetry's resolver — while correct — is known for fetching far more metadata than it strictly needs before making a decision.

rough install time comparison — large project (~60 deps)
# cold cache, fresh venv
pip install -r requirements.txt   ~45s
poetry install                    ~30s  (from lock file)
uv sync                           ~4s   (cold)
uv sync                           <0.5s (warm cache, no changes)
pip + venvSequential fetches. No global cache by default. Adequate for most use cases, slow in CI without explicit caching.
PoetryFaster than pip from a lock file, but slow resolution phase. The --no-root flag helps in CI.
uv10–100× faster than pip. Parallel network I/O, global wheel cache, zero-copy hardlinks on supported filesystems.
condaSlow SAT solver. Fetching repodata (full channel metadata) is expensive. mamba (a C++ reimplementation) is much faster.

These two requirements both sit outside the boundary of what pure Python package managers can address — and understanding why clarifies when you actually need conda or pyenv.

Non-Python dependencies are C/C++ shared libraries, CUDA toolkits, Fortran runtimes, R packages, and other OS-level binaries. A Python package like numpy ships a pre-compiled wheel that bundles most C code inside it (thanks to auditwheel / delocate), so you don't need BLAS separately for numpy. But packages like gdal, GPU-accelerated code, or anything that links against system libraries at runtime do need those libraries present on the system — and pip/uv have no idea they exist. conda solves this by maintaining its own binary package ecosystem, separate from PyPI, where packages like cudatoolkit and libgdal are first-class citizens.

Python version management is the question of "which Python is running this code". pip and Poetry assume the answer is already settled — they install into a Python, they don't manage which Python that is. Historically pyenv (or pyenv-win) solved this by compiling multiple Python versions side by side. uv now bundles this natively: uv python install 3.13 downloads and installs a managed Python build, and uv run --python 3.11 script.py uses it without any shell gymnastics.

when you need conda

CUDA / GPU packages, geospatial (GDAL, PROJ), bioinformatics (samtools, bowtie), anything that links against non-bundled C libraries at runtime.

when uv replaces pyenv

Pure Python projects that just need to switch between interpreter versions. uv python install + .python-version file covers most developer workflows.

pip + venvNo Python version management. No non-Python dep support. You need pyenv + system package manager alongside it.
PoetryCan use a pyenv-managed Python, but doesn't install Python itself. No non-Python dep support.
uvManages Python versions natively. Still no non-Python dep support — for those, use conda or a system package manager.
condaManages both the Python interpreter and non-Python libraries in a unified env. The right tool when you genuinely need the cross-language package graph.
