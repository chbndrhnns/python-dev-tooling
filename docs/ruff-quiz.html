A linter reads source code and reports violations of rules without changing any files. Rules detect everything from actual bugs (using a variable before assignment) to style issues (unused imports) to security antipatterns (use of assert in production code).

Ruff ships with over 800 rules, each with a short code like E501 (line too long), F401 (unused import), or B006 (mutable default argument). Rules are grouped into rule sets identified by a prefix letter — E/W for pycodestyle, F for Pyflakes, B for flake8-bugbear, I for isort, and many more.

You opt into rules with select and opt out with ignore. A specific violation on a line can be suppressed with a # noqa: F401 comment. Ruff can also auto-fix many violations — adding missing imports, removing unused ones, rewriting deprecated patterns — via ruff check --fix.

bad code → ruff output
# bad.py
import os          # F401 — imported but unused
import sys, json   # E401 — multiple imports on one line

def greet(name, items=[]):   # B006 — mutable default argument
    x = 1              # F841 — local variable assigned but never used
    print("hello " + name)

$ ruff check bad.py
bad.py:1:1: F401 `os` imported but unused
bad.py:2:1: E401 Multiple imports on one line
bad.py:4:18: B006 Do not use mutable data structures for argument defaults
bad.py:5:5: F841 Local variable `x` is assigned but never used
pyproject.toml — rule selection
[tool.ruff.lint]
select = ["E", "F", "B", "I"]   # enable rule sets
ignore = ["E501"]               # suppress line-length rule
fixable = ["ALL"]               # auto-fix everything possible

A formatter rewrites your source files to enforce a consistent style — indentation, spacing, quote style, trailing commas, line wrapping. Unlike a linter it doesn't report violations; it just fixes them. The result is deterministic: the same input always produces the same output.

Ruff's formatter is a Black-compatible drop-in replacement, roughly 35× faster. "Compatible" means it produces the same output as Black on virtually all real-world Python code, so you can switch without changing your diff history. The key difference: ruff handles both linting and formatting in one tool, eliminating the friction of coordinating Black + isort + flake8 separately.

The formatter is intentionally opinionated — very few knobs. You can configure line-length, quote-style (single vs. double), indent-style (spaces vs. tabs), and whether to add a magic trailing comma. That's mostly it. The philosophy is that fewer options means less team debate.

before → after ruff format
# before
def foo(a,b,c):
    return   a+b+c
x = {'key' :   'value'  , 'other':'thing'}
if (x == True):
    pass

# after
def foo(a, b, c):
    return a + b + c


x = {"key": "value", "other": "thing"}
if x == True:
    pass
pyproject.toml — formatter options
[tool.ruff.format]
quote-style           = "double"   # default, Black-compatible
indent-style          = "space"
skip-magic-trailing-comma = false  # respect trailing commas
line-ending           = "auto"

Ruff is configured via pyproject.toml (under [tool.ruff]) or a standalone ruff.toml. Configuration is hierarchical: a base config can be extended by sub-configs, and rules can be overridden per file glob.

per-file-ignores is the most important scoping feature — it lets you suppress specific rules only in certain paths. The canonical use case is test files: assert is banned in production code (S101) but necessary in pytest. You also often want to allow unused imports in __init__.py files where re-exporting is intentional.

target-version tells ruff which Python version your code needs to support, which affects which rules fire and how auto-fixes are generated — for example, using f"..." instead of ".format()" is only safe to auto-fix if you're targeting Python 3.6+.

pyproject.toml — per-file overrides
[tool.ruff]
line-length    = 100
target-version = "py311"

[tool.ruff.lint]
select = ["E", "F", "B", "S", "I"]

[tool.ruff.lint.per-file-ignores]
"tests/**"     = ["S101"]         # allow assert in tests
"__init__.py"  = ["F401"]         # allow re-exports
"scripts/**"   = ["T201"]         # allow print() in scripts

Ruff is designed to be fast enough that it can run on every keystroke in an editor or on every commit in CI without adding perceptible latency. It has first-class support for the Language Server Protocol (LSP) via ruff server, which powers the VS Code extension and any other LSP-compatible editor (Neovim, Helix, Zed).

In CI, the key distinction is check mode vs. fix mode. In CI you want ruff check . and ruff format --check . — both exit with a non-zero code if violations exist, without modifying files. Local developer workflows use --fix and ruff format to apply changes in place.

The pre-commit integration is straightforward: ruff provides an official hook at github.com/astral-sh/ruff-pre-commit. Because ruff is a single binary with no Python dependencies, it installs once and runs in milliseconds — much faster than managing a separate flake8 + black + isort + pyupgrade hook chain.

.pre-commit-config.yaml

- repo: https://github.com/astral-sh/ruff-pre-commit
hooks: ruff (lint + fix) and ruff-format (format). Replaces black + flake8 + isort hooks.

CI check (no modifications)

ruff check . — exits 1 if violations found.
ruff format --check . — exits 1 if formatting needed. Both are idempotent and safe to run in parallel.
